# Roblox Route Bot

Внешняя (без инъекции в игру) программа на C#/WPF: рисуешь маршрут — персонаж бежит по нему,
с **визуальной обратной связью** против дрейфа. Управление через `SendInput` (**только WASD**,
камеру/мышь бот не трогает), позиция считывается **с экрана** компьютерным зрением. Путь привязан
к **миру**: повернул камеру — маршрут остаётся на том же месте на земле. В игру ничего не встраивается.

> Анти-чит Hyperion/Byfron ловит инъекции и чтение памяти процесса. Здесь ни того, ни другого:
> только ввод уровня ОС + чтение картинки с экрана (WGC/GDI). Но: (1) это против ToS Roblox;
> (2) у игры может быть **серверный** детект ботоподобного движения. Риск бана реален — на твой страх.

## Главная идея (top-down зрение)

Камеру в игре ставишь **строго сверху** (bird's-eye). Тогда экран ≈ 2D-карта: персонаж в центре,
а при ходьбе весь мир жёстко «скроллится» под ним. Бот меряет этот скролл и понимает, куда и
насколько уехал персонаж — это и есть честный closed-loop, который не врёт «я дошёл».

> Честно: Roblox не даёт идеальных 90° сверху без executor'а (= инъекция = бан) — реально
> получается косой вид. Это лечится **ползунком «наклон камеры»**: бот разворачивает косой кадр
> в виртуальный top-down (IPM-ректификация), и зрение работает как при идеальном виде сверху.
> Комфортно до ~40° наклона. На ровной земле — отлично; на склонах/высоких объектах точность падает.
> Дрейф **ограничен** ориентирами, но не обнулён полностью.

## Запуск

```powershell
.\run.bat
```
(собирает Release под x64 и запускает; exe в `bin\x64\Release\net8.0-windows10.0.19041.0\`)

## Как пользоваться

1. **Окно игры** — кнопка «Скан», выбери окно Roblox.
2. **Камера сверху** — подними камеру как можно вертикальнее. Если идеально сверху не выходит —
   выстави ползунок **«наклон камеры»** примерно под свой угол и подкрути по телеметрии (signal вверх,
   drift вниз). FOV по умолчанию 70.
3. **Лайв-превью** — включи тумблер: в канвасе появится картинка игры + панель телеметрии.
4. **Нарисуй маршрут** — зажми ЛКМ поверх картинки и веди. Линия = путь по карте.
5. **Режим «Зрение»** (по умолчанию) — closed-loop по экрану.
6. **Запустить** — отсчёт 2 сек, **кликни в окно игры** (ввод идёт в активное окно). Дальше бот
   сам коротко жмёт W и D — это **авто-калибровка** базиса (учится, куда едет перс на экране),
   затем идёт по маршруту.
7. **Кружок-маркер** — во время прогона можно **перетащить** его мышью на реальную позицию
   персонажа (ручной ре-якорь, если зрение уползло).
8. **Стоп** — остановить, все клавиши отпускаются.

## Режимы «глаз»

| Режим | Что делает | Дрейф |
|---|---|---|
| **Зрение** (по умолчанию) | поза (поворот+сдвиг) по Fourier-Mellin + ориентиры (closed-loop) | ограничен ориентирами |
| **Поток** | старый блок-матчинг одного патча | ползёт |
| **Open** | слепое воспроизведение по таймингу (база) | копится |

- **Зрение:** два слоя. Слой 1 — **Fourier-Mellin**: лог-полярная фазовая корреляция даёт МЕЖКАДРОВЫЙ
  ПОВОРОТ камеры, де-ротация + фазовая корреляция дают сдвиг. Поза копится → путь держится привязанным
  к миру даже при повороте камеры. Слой 2 — бот **запоминает контрастные пиксельные патчи** (ориентиры,
  поворото-устойчивый NCC) с их мировой позицией и, узнавая снова, подтягивает оценку → гасит дрейф.
  Параметр `px → world` — множитель чувствительности (1.0 = авто-масштаб от размера канваса).

## Архитектура

```
[UnifiedCaptureSource: WGC/GDI] → [VisualOdometryProvider] → [PurePursuitController] → [MovementMapper(базис)] → [InputSender]
        |                              |  Layer1: FourierMellinEstimator (поворот+сдвиг, FFT)
        |                              |  Layer2: LandmarkMap (поворото-устойчивый NCC)
        |                              └─ FrameTransform (единая система координат + базис WASD + heading)
        └──────────── BotEngine: отсчёт → WasdCalibrator (W/D) → цикл ~60 Гц ────────────┘
```

- `Vision/` — захват (`ICaptureSource`/`UnifiedCaptureSource`, `WgcCapture`, `ScreenCapture`,
  `RectifyingCaptureSource`+`InversePerspectiveMap` — IPM-ректификация наклона),
  FFT-ядро (`FourierProcessor`, параллельный), оценщики (`IPoseEstimator`→`FourierMellinEstimator` поза;
  `IMotionEstimator`→`PhaseCorrelationEstimator`/`SadBlockEstimator` сдвиг), `ImageWarp` (поворот/
  лог-полярность), `LandmarkMap` (гашение дрейфа), `FrameTransform` (координаты/базис/heading),
  провайдеры (`VisualOdometryProvider` — основной, `OpticalFlowProvider`, `DeadReckoningProvider`).
- `Core/` — `PurePursuitController` (следование), `MovementMapper` (направление→WASD через базис),
  `WasdCalibrator` (авто-калибровка), `BotEngine` (цикл + защита от застревания/потери).
- `Input/` — `SendInput` скан-кодами, высокоточный таймер. `Native/` — окна.

## Честные ограничения

- Не настоящий top-down, а ~80° косой вид → перспектива; масштаб точен лишь для ровной земли.
- На однородной/повторяющейся текстуре (трава, вода, плитка) и фазовая корреляция, и NCC слепнут —
  бот «держит» позицию и копит дрейф, пока сигнал не вернётся.
- Дрейф **ограничен**, но не обнулён: между ориентирами позиция «пилит». Глобальная оптимизация
  карты (pose-graph) — вне v1.
- `SendInput` доходит, только пока **Roblox в фокусе** (нельзя alt-tab'нуться во время прогона).
- **Поворот камеры (yaw)** отслеживается (Fourier-Mellin), **наклон (pitch)** — выправляется
  ректификацией по ползунку (комфортно до ~40°). НО: зум камеры НЕ моделируется; склоны/холмы ломают
  плоскостную ректификацию (одна плоскость земли); резкий рывок камеры за кадр может сбить угол.
- Скорость: FFT распараллелен по P-ядрам (~6 мс/тик, ~160 fps). GPU не используется намеренно — на
  кадрах 256×256 оверхед запуска CUDA-ядра больше пользы, плюс это нативка (риск на чужой машине).
- Динамика на экране (другие игроки, NPC, анимация) портит глобальную оценку — отсекается фильтрами и
  ориентирами, но в людной/однотекстурной сцене точность падает (тогда поможет ручной ре-якорь маркером).
